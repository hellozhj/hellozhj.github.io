一. 小程序工程目录

1.0 新建项目

首先下载微信开发者工具：https://developers.weixin.qq.com/miniprogram/dev/devtools/download.html

打开工具，创建小程序项目：

AppID是什么？怎么获取？

AppID(小程序ID)跟人的身份证号一样，这个是小程序的身份证明。是唯一的，不会跟其他的程序重复，我们很多时候需要用到AppID（如小程序上传），那么我们如何获取AppID呢？

1.进入小程序微信公众平台网址：https://mp.weixin.qq.com/

2.扫码登录或选择账号密码登录

3.在开发管理>开发设置查看自己的AppID

项目目录：

我们通过开发者工具快速创建了一个 QuickStart 项目。你可以留意到这个项目里边生成了不同类型的文件:
  1. .json 后缀的 JSON 配置文件
  2. .wxml 后缀的 WXML 模板文件
  3. .wxss 后缀的 WXSS 样式文件
  4. .js 后缀的 JS 脚本逻辑文件

接下来我们分别看看这4种文件的作用。

1.1 JSON 配置

JSON 是一种数据格式，并不是编程语言，在小程序中，JSON扮演静态配置的角色。

我们可以看到在项目的根目录有一个 app.json 和 project.config.json，此外在 pages/index 目录下还有一个 index.json，我们依次来说明一下它们的用途。

1.1.1 app.json

app.json 是当前小程序的全局配置，包括了小程序的所有页面路径、界面表现、网络超时时间、底部 tab 等。QuickStart 项目里边的 app.json 配置内容如下：

{
  "pages": ["pages/index/index"],
  "window": {
    "backgroundTextStyle": "light",
    "navigationBarBackgroundColor": "#fff",
    "navigationBarTitleText": "Weixin",
    "navigationBarTextStyle": "black"
  },
  "style": "v2",
  "sitemapLocation": "sitemap.json"
}
1) pages字段

用于描述当前小程序所有页面路径，这是为了让微信客户端知道当前你的小程序页面定义在哪个目录，初始化项目后默认是在pages文件夹下。

当我们添加数组项时，编辑器会帮我们自动新建目录：

"pages": ["pages/index/index", "pages/list/index"]

也可以通过在目录上直接右键新建文件夹并新建page来创建新页面，这个时候app.json的pages字段也会自动添加新增页面的路径。

tips：怎么禁用微信开发者工具的彩虹括号？在settings.json中添加以下代码：

"bracket-pair-colorizer-2.excludedLanguages": [
  ".wxml",
  ".wxss",
  ".js",
  ".json"
],
"bracket-pair-colorizer-2.colors": []
2) window字段

https://developers.weixin.qq.com/miniprogram/dev/reference/configuration/app.html#window

  • backgroundTextStyle：下拉 loading 的样式，值为string类型，仅支持 dark / light。单独写看不出什么效果，因为这个配置的意思是进行下拉刷新的时候的样式。但问题是，小程序默认是关闭下拉刷新功能的，这时候就需要我们手动往window对象中添加：

    "enablePullDownRefresh": true // 是否开启当前页面下拉刷新，默认为false。不建议全局开启，不是所有页面都需要下拉刷新的功能。我们可以在page.json定义，对每一个页面进行配置。

3) style字段

只有一个合法值：v2。微信小程序使用的ui框架是we ui，v2表示第二版样式。如果不想使用第二版样式，只能直接将style字段删除，不能传入v1。

4) sitemapLocation字段

指明 sitemap.json 的位置；默认为 ‘sitemap.json’ 即在 app.json 同级目录下名字的 sitemap.json 文件。微信现已开放小程序内搜索，开发者可以通过 sitemap.json 配置，或者管理后台页面收录开关来配置其小程序页面是否允许微信索引。当开发者允许微信索引时，微信会通过爬虫的形式，为小程序的页面内容建立索引。当用户的搜索词条触发该索引时，小程序的页面将可能展示在搜索结果中。

5) tabBar字段

https://developers.weixin.qq.com/miniprogram/dev/reference/configuration/app.html#tabBar

app.json中默认没有tabBar字段，但是该字段很重要。如果小程序是一个多 tab 应用（客户端窗口的底部或顶部有 tab 栏可以切换页面），可以通过 tabBar 配置项指定 tab 栏的表现，以及 tab 切换时显示的对应页面。

另外，有一个list字段，用来表示tab 的列表，list 接受一个数组，只能配置最少 2 个、最多 5 个 tab。tab 按数组的顺序排序，每个项都是一个对象，其属性值如下：

}

1.1.2 project.config.json

通常大家在使用一个工具的时候，都会针对各自喜好做一些个性化配置，例如界面颜色、编译配置等等，当你换了另外一台电脑重新安装工具的时候，你还要重新配置。

考虑到这点，小程序开发者工具在每个项目的根目录都会生成一个 project.config.json，你在工具上做的任何配置都会写入到这个文件，当你重新安装工具或者换电脑工作时，你只要载入同一个项目的代码包，开发者工具就自动会帮你恢复到当时你开发项目时的个性化配置，其中会包括编辑器的颜色、代码上传时自动压缩等等一系列选项。

1.1.3 page.json

这里的 page.json 其实用来表示 pages/index 目录下的 index.json 这类和小程序页面相关的配置。

如果你整个小程序的风格是蓝色调，那么你可以在 app.json 里边声明顶部颜色是蓝色即可。实际情况可能不是这样，可能你小程序里边的每个页面都有不一样的色调来区分不同功能模块，因此我们提供了 page.json，让开发者可以独立定义每个页面的一些属性，例如刚刚说的顶部颜色、是否允许下拉刷新等等。所有配置项都写在一起，没有单独的window属性。

其他配置项细节可以参考文档 页面配置 。

1.2 WXML 模板

WXML 充当的就是类似 HTML 的角色。往往写 HTML 的时候，经常会用到的标签是 div, p, span，小程序中没有这些组件，取而代之的是view，text等基础组件和其他封装过的组件，如表单组件、导航组件、媒体组件等。模板有数据绑定、列表渲染、条件渲染、模板、引用等功能。

1.3 WXSS 样式

WXSS 具有 CSS 大部分的特性，小程序在 WXSS 也做了一些扩充和修改。

  1. 尺寸单位为rpx（responsive pixel）: 可以根据屏幕宽度进行自适应。规定屏幕宽为750 rpx。如在 iPhone6 上，屏幕宽度为375 px，共有750个物理像素，则750 rpx = 375 px = 750物理像素，1 rpx = 0.5 px = 1物理像素。

    建议：小程序设计稿宽度为750 px，这样子1 px就等于1 rpx，不用换算。

  2. 提供了全局样式和局部样式。和前边 app.json, page.json 的概念相同，你可以写一个 app.wxss 作为全局样式，会作用于当前小程序的所有页面，局部页面样式 page.wxss 仅对当前页面生效。对同一个选择器，页面样式表比公共样式表拥有更高的权重。

    /* app.wxss */
view {
  color: red;
}

/* page.wxss */
view {
  color: blue;
}
    <!-- view元素字体为蓝色 -->
<view>hello world</view>

  3. 此外 WXSS 仅支持部分 CSS 选择器

更详细的文档可以参考 WXSS 。

1.4 JS 逻辑交互

提供了全局的小程序逻辑app.js和针对单个页面的逻辑page.js。

每个小程序都需要在 app.js 中调用 App 方法注册小程序实例（必须调用且只能调用一次。不然会出现无法预期的后果），并可以在其中去绑定生命周期回调函数、设置全局数据globalData等。而在page.js中调用Page方法注册小程序中的一个页面，接受一个 Object 类型参数，其指定页面的初始数据、生命周期回调、事件处理函数等。

1.5 注册小程序

每个小程序都需要在 app.js 中调用 App 方法注册小程序实例，绑定生命周期回调函数、错误监听和页面不存在监听函数等。

// app.js
App({
  onLaunch() {
    // 监听小程序初始化，全局只触发一次
    console.log("onLaunch");
  },
  onShow() {
    // 监听小程序启动或切前台
    console.log("onShow");
    console.log(undefined.length) // 会触发下面的onError
  },
  onHide() {
    // 监听小程序切后台 
    console.log("onHide");
  },
  onError(err) {
    // 小程序发生脚本错误或 API 调用报错时触发
    console.log(err);
  },
  globalData: {
    num: 1,
    string: "你好",
    boolean: true,
  },
});

整个小程序只有一个 App 实例，是全部页面共享的。开发者可以通过 getApp 方法获取到全局唯一的 App 实例，获取 App 上的数据或调用开发者注册在 App 上的函数。

// pages/index/index.js
const app = getApp();
console.log(app);
console.log(app.globalData);

app.globalData的变化没法动态响应在视图上，一般用来存放一些不需要变动的数据。

1.6 注册页面

对于小程序中的每个页面，都需要在页面对应的 js 文件中调用Page方法注册，指定页面的初始数据、生命周期回调、事件处理函数等。

1.6.1 页面初始数据

data 是页面第一次渲染使用的初始数据。

页面加载时，data 将会以JSON字符串的形式由逻辑层传至渲染层，因此data中的数据必须是可以转成JSON的类型：字符串，数字，布尔值，对象，数组。

渲染层可以通过 WXML 对数据进行绑定。

1.6.2 页面生命周期

先跳过，参照第八节页面生命周期。

1.6.3 页面事件处理函数

1.6.3.1 onPullDownRefresh()

监听用户下拉刷新事件：

  • 需要在app.json的window选项中或页面配置中开启enablePullDownRefresh。

    // pages/index/index.json
{
  "enablePullDownRefresh": true
}
    // pages/index/index.js
onPullDownRefresh() {
  console.log("下拉刷新");
  // 控制什么时候结束下拉刷新的样式
  setTimeout(() => {
    wx.stopPullDownRefresh({
      success: (res) => {
        console.log("已更新");
      },
    });
  }, 2000);
},

  • 可以通过wx.startPullDownRefresh触发下拉刷新，调用后触发下拉刷新动画，效果与用户手动下拉刷新一致。注意：调用wx.startPullDownRefresh的时候，也会同时触发onPullDownRefresh方法。

    <!-- pages/index/index.wxml -->
<button bindtap="handleRefresh">点击进行刷新</button>
    // pages/index/index.js
Page({
  handleRefresh() {
    wx.startPullDownRefresh({
      success: (res) => {
        console.log("开始下拉刷新");
      },
    });
  },
});

  • 当处理完数据刷新后，wx.stopPullDownRefresh可以停止当前页面的下拉刷新。

    <!-- pages/index/index.wxml -->
<button bindtap="handleRefresh">点击进行刷新</button>
    // pages/index/index.js
Page({
  data: {
    flag: false,
  },
  onPullDownRefresh() {
    console.log("下拉刷新");
    setTimeout(() => {
      console.log('获取到数据')
      wx.stopPullDownRefresh();
    }, 2000);
  },
  handleRefresh() {
    wx.startPullDownRefresh();
  },
});
1.6.3.2 onReachBottom()

监听用户上拉触底事件。

  • 可以在app.json的window选项中或页面配置中设置触发距离onReachBottomDistance。
  • 在触发距离内滑动期间，本事件只会被触发一次。
// pages/index/index.json
// onReachBottomDistance：页面上拉触底事件触发时距页面底部距离，单位为px，默认值为50。
{
  "onReachBottomDistance": 50 
}
// pages/index/index.js
Page({
  onReachBottom() {
    console.log("触底了");
  },
});

二. 数据绑定

WXML 中的动态数据均来自对应 Page 的 data。

数据绑定使用 Mustache 语法（双大括号）将变量包起来，可以用于以下场景：

2.1 页面内容渲染

<view>{{msg}}</view>
Page({
  data: {
    msg: "hello world",
  },
});

2.2 组件属性

<view id="{{'item-'+id}}" data-id="item-{{id}}">{{msg}}</view>
Page({
  data: {
    msg: "hello world",
    id: 0,
  },
});

2.3 控制属性

<view wx:if="{{show}}">{{msg}}</view>
Page({
  data: {
    msg: "hello world",
    show: false,
  },
});

2.4 关键字

true：boolean 类型的 true，代表真值。

false： boolean 类型的 false，代表假值。

<checkbox checked="{{checked}}"></checkbox>
Page({
  data: {
    checked: false,
  },
});

上面的写法等同于：

<checkbox checked="{{false}}"></checkbox>

特别注意：如果直接写 checked=”false”，其计算结果是一个字符串，转成 boolean 类型后代表真值。

2.5 三元运算和逻辑判断

<view wx:if="{{a > 1 ? true : false}}">hello world</view>
<view wx:if="{{a > 1}}">hello world</view>
Page({
  data: {
    a: 1,
  },
});

2.6 数据路径运算

<view>{{obj.key}}</view>
<view>{{arr[0]}}</view>
Page({
  data: {
    obj: {
      key: "Hello ",
    },
    arr: [1],
  },
});

三. 事件

  • 事件是视图层到逻辑层的通讯方式。
  • 事件可以将用户的行为反馈到逻辑层进行处理。
  • 事件可以绑定在组件上，当达到触发事件，就会执行逻辑层中对应的事件处理函数。
  • 事件对象可以携带额外信息，如 id, dataset。

3.1 事件分类

事件分为冒泡事件和非冒泡事件：

  1. 冒泡事件：当一个组件上的事件被触发后，该事件会向父节点传递。
  2. 非冒泡事件：当一个组件上的事件被触发后，该事件不会向父节点传递。

WXML的冒泡事件列表

3.2 普通事件绑定

我们可以用bind来绑定普通事件。

我们先来看小程序的单击事件bindtap，语法是bind后面跟事件类型tap，除了这么写，还可以写bind:tap（自基础库版本 1.5.0 起，在大多数组件和自定义组件中， bind 后可以紧跟一个冒号，其含义不变，如 bind:tap 。基础库版本 2.8.1 起，在所有组件中开始提供这个支持。）。

我们在组件中绑定一个事件处理函数，在相应的Page定义中写上相应的事件处理函数。

<button bind:tap="handleTap">tap me</button>
Page({
  // 不需要和Vue一样将方法写在methods中
  handleTap() {
    console.log("点击了");
  },
});

3.3 基础事件对象

当组件触发事件时，逻辑层绑定该事件的处理函数会收到一个基础事件对象，我们可以用event来接收。

事件对象有两个比较重要的属性currentTarget和target，他们都可以用来接收一些属性值集合。

<button foo="bar" id="{{'item-'+id}}" data-id="{{'item-'+id}}" data-foo="bar" bind:tap="handleTap">tap me</button>
Page({
  data: {
    id: 0,
  },
  handleTap(e) {
    console.log(e);
  },
});

在e.target和e.currentTarget里面，我们可以通过id字段获取到id属性的值（只能获取到id字段，其他自定义字段获取不到），也可以通过dataset.xx获取到data-xx的值。

currentTarget和target的不同：

.child {
  width: 150rpx;
  height: 150rpx;

Page({

    id: 0,

四. setData

4.1 语法

由于小程序的逻辑层和视图层是两个独立的运行环境、分属不同的线程或进程，不能直接进行数据共享，需要进行数据的序列化、跨线程/进程的数据传输、数据的反序列化，因此数据传输过程是异步的、非实时的。

直接修改 this.data 而不调用 this.setData 是无法改变页面的状态的。

<!--index.wxml-->
<view>{{num}}</view>
<button bindtap="handleChangeNum">change num</button>
// index.js

Page({
  data: {
    num: 1,
  },
  handleChangeNum() {
    this.data.num = 2;
    console.log(this.data.num);
  },
});

setData 函数用于改变对应的 this.data 的值（修改数据是同步），同时将数据从逻辑层发送到视图层（视图更新是异步）。setData接收两个参数，第一个参数为Object类型，表示这次要修改的数据；第二个参数为setData引起的界面更新渲染完毕后的回调函数，非必传，了解即可。

注意：

  • 直接修改 this.data 而不调用 this.setData 是无法改变页面的状态的，因为只有setData可以触发视图更新。

  • Object 以 key: value 的形式表示，将 this.data 中的 key 对应的值改变成 value。

    其中 key 可以以数据路径的形式给出，支持改变数组中的某一项或对象的某个属性，如 array[2].message，a.b.c.d，并且不需要在 this.data 中预先定义。

<view>{{text}}</view>
<button bindtap="changeText"> Change normal data </button>
<view>{{num}}</view>

4.2 数据双向绑定

我们可以自己实现数据双向绑定（类似于Vue的:value和@input）：

<input type="text" value="{{inputVal}}" bindinput="handleInput" />
<view>------</view>
<view>{{inputVal}}</view>
Page({
  data: {
    inputVal: "hello",
  },
  handleInput(e) {
    console.log(e);
    this.setData({
      inputVal: e.detail.value,
    });
    console.log(this.data.inputVal);
  },
});

除了绑定一个字段，还可以绑定data路径：

<input type="text" value="{{obj.msg}}" bindinput="handleInput" />
<view>------</view>
<view>{{obj.msg}}</view>

  data: {
    obj: {
      msg: "hello",
    },
  },
  handleInput(e) {
    console.log(e);
    this.setData({
      "obj.msg": e.detail.value,
    });
    console.log(this.data.obj.msg);
  },
});

4.3 合理使用setData

setData 是小程序开发中使用最频繁、也是最容易引发性能问题的接口。

setData 的过程，大致可以分成几个阶段：

  • 逻辑层虚拟 DOM 树的遍历和更新，触发组件生命周期和 observer 等；
  • 将 data 从逻辑层传输到视图层；
  • 视图层虚拟 DOM 树的更新、真实 DOM 元素的更新并触发页面渲染更新。

每次 setData 都会触发逻辑层虚拟 DOM 树的遍历和更新，也可能会导致触发一次完整的页面渲染流程。

使用建议：

  1. data 应只包括和渲染相关的数据。

    setData 应只用来进行渲染相关的数据更新。用 setData 的方式更新渲染无关的字段，会触发额外的渲染流程，或者增加传输的数据量，影响渲染耗时：

    const HOURS = 24; // good

Page({
  data: {
    hours: 24, // bad
  },
});

  2. 对连续的 setData 调用尽可能的进行合并。

    // bad
this.setData({
  flag: true,
});
if (this.data.flag) {
  this.setData({
    // ...
  });
}
    // good
let flag = true;
if (flag) {
  this.setData({
    flag,
    // ...
  });
}

  3. setData 应只传发生变化的数据。setData 应只传入发生变化的字段，建议以数据路径形式改变数组中的某一项或对象的某个属性，如：

    <view>{{arr[0].msg}}</view>
<view>{{obj.person.name}}</view>
<button bindtap="handleChangeData">click to change data</button>
    Page({
  data: {
    arr: [{ msg: "hello" }],
    obj: {
      person: {
        name: "jojo",
      },
    },
  },
  handleChangeData() {
    this.setData({
      "arr[0].msg": "goodbye",
      "obj.person.name": "rose",
    });
  },
});

五. 列表渲染

5.1 wx:for

在组件上使用 wx:for 控制属性绑定一个数组，即可使用数组中各项的数据重复渲染该组件。

默认数组的当前项的下标变量名默认为 index，数组当前项的变量名默认为 item。

<view wx:for="{{person}}">{{index}} : {{item.name}}</view>
Page({
  data: {
    person: [{ name: "jojo" }, { name: "rose" }, { name: "jack" }],
  },
});

使用 wx:for-item 可以指定数组当前元素的变量名，

使用 wx:for-index 可以指定数组当前下标的变量名：

<view wx:for="{{person}}" wx:for-item="ele" wx:for-index="i">{{i}} : {{ele.name}}</view>

Vue的写法：

<template>
  <div id="app">
    <div v-for="(item, index) in person" :key="item.name">
      {{ index }}:{{ item.name }}
    </div>
  </div>
</template>

<script>
export default {
  data() {
    return {
      person: [{ name: "jojo" }, { name: "rose" }, { name: "jack" }],
    };
  },
};
</script>

可以将 wx:for 用在<block/>标签上，以渲染一个包含多节点的结构块。

注意： <block/> 并不是一个组件，它仅仅是一个包装元素，不会在页面中做任何渲染，只接受控制属性。

<block wx:for="{{person}}">
  <view>
    {{index}} : {{item.name}}
  </view>
</block>

    person: [{ name: "jojo",id:1 }, { name: "rose",id:2 }, { name: "jack",id:3 }],

});

5.4 wx:key

如果列表中项目的位置会动态改变或者有新的项目添加到列表中，并且希望列表中的项目保持自己的特征和状态（如 input 中的输入内容，switch 的选中状态），需要使用 wx:key 来指定列表中项目的唯一的标识符。

wx:key 的值以两种形式提供：

  1. 字符串，代表在 for 循环的 array 中 item 的某个 property，该 property 的值需要是列表中唯一的字符串或数字，且不能动态改变，例如id。
  2. 保留关键字 *this 代表在 for 循环中的 item 本身，这种表示需要 item 本身是一个唯一的字符串或者数字。

当数据改变触发渲染层重新渲染的时候，会校正带有 key 的组件，框架会确保他们被重新排序，而不是重新创建，以确保使组件保持自身的状态，并且提高列表渲染时的效率。

如不提供 wx:key，会报一个 warning， 如果明确知道该列表是静态，或者不必关注其顺序，可以选择忽略。

六. 条件渲染

6.1 wx:if

在框架中，使用 wx:if="" 来判断是否需要渲染该代码块：

  data: {
    flag: false,
  },

6.2 block wx:if

因为 wx:if 是一个控制属性，需要将它添加到一个标签上。如果要一次性判断多个组件标签，可以使用一个 <block/> 标签将多个组件包装起来，并在上边使用 wx:if 控制属性。注意： <block/> 并不是一个组件，它仅仅是一个包装元素，不会在页面中做任何渲染，只接受控制属性。

<block wx:if="{{flag}}">
  <view>hello world</view>
  <view>goodbye world</view>
</block>
Page({
  data: {
    flag: false,
  },
});

6.3 wx:elif，wx:else

也可以用 wx:elif 和 wx:else 来添加一个 else 块：

<view wx:if="{{score >= 90}}">优秀</view>
<view wx:elif="{{score >= 60 && score < 90}}">还行</view>
<view wx:else>差</view>
Page({
  data: {
    score: 75,
  },
});

6.4 wx:if vs hidden

因为 wx:if 之中的模板也可能包含数据绑定，所以当 wx:if 的条件值切换时，框架有一个局部渲染的过程，因为它会确保条件块在切换时销毁或重新渲染。

同时 wx:if 也是惰性的，如果在初始渲染条件为 false，框架什么也不做，在条件第一次变成真的时候才开始局部渲染。

相比之下，hidden 就简单的多，组件始终会被渲染，只是简单的控制显示与隐藏。

一般来说，wx:if 有更高的切换消耗而 hidden 有更高的初始渲染消耗。因此，如果需要频繁切换的情景下，用 hidden 更好，如果在运行时条件不大可能改变则 wx:if 较好。

注意：hidden和display属性冲突。

八.页面生命周期

  • onLoad(Object query)
    页面加载时触发。一个页面只会调用一次，可以在 onLoad 的参数中获取打开当前页面路径中的参数。
  • onShow()
    页面显示/切入前台时触发。
  • onReady()
    页面初次渲染完成时触发。一个页面只会调用一次，代表页面已经准备妥当，可以和视图层进行交互。
    注意：对界面内容进行设置的 API 如wx.setNavigationBarTitle，请在onReady之后进行。
  • onHide()
    页面隐藏/切入后台时触发。 如 wx.navigateTo 或底部 tab 切换到其他页面，小程序切入后台等。
  • onUnload()
    页面卸载时触发。如wx.redirectTo、wx.navigateBack、wx.reLaunch到其他页面时。

初始进入页面时，触发的顺序是onLoad > onShow > onReady。但是，这并不意味着，onLoad中的所有逻辑执行完后，才会去触发其他生命周期。

十. WXS语法

WXS（WeiXin Script）是小程序的一套脚本语言，结合 WXML，可以构建出页面的结构。

WXS 与JavaScript是不同的语言，有自己的语法，并不和 JavaScript 一致。

10.1 WXS 有什么用？

微信为何要脱离 JavaScript ，单独创造一套语言呢？这要从微信小程序的底层逻辑（运行环境）讲起。

小程序的运行环境分为逻辑层和视图层，分别由2个线程管理，其中：
WXML 模板和 WXSS 样式工作在视图层，界面使用 WebView 进行渲染
JavaScript代码工作在逻辑层，运行在JsCore或v8里
小程序在视图层与逻辑层两个线程间提供了数据传输和事件系统。这样的分离设计，带来了显而易见的好处：
逻辑和视图分离，即使业务逻辑计算非常繁忙，也不会阻塞渲染和用户在视图层上的交互。
但同时也带来了明显的坏处：
视图层（WebView ）中不能运行JS，而逻辑层JS又无法直接修改页面DOM，数据更新及事件系统只能靠线程间通讯，但跨线程通信的成本极高，特别是需要频繁通信的场景。

从本质来讲，WXS是一种被限制过的、运行在视图层webview里的js。它并不是真的发明了一种新语言。

WXS具备如下特征：
WXS是可以在视图层（webview）中运行的JS。
WXS是被限制过的JavaScript，可以进行一些简单的逻辑运算
故可以得出WXS的适用场景，主要包括：
数据格式处理，比如文本、日期格式化，或者国际化。
类比Vue，微信小程序的WXS语法 可以当做Vue的计算属性和Vue filter过滤器使用。因为WXS中的函数可以写在{{}}中 。

10.2 WXS 模块

WXS 代码可以编写在 wxml 文件中的 <wxs> 标签内，或以 .wxs 为后缀名的文件内。

每一个 .wxs 文件和 <wxs> 标签都是一个单独的模块。每个模块都有自己独立的作用域。即在一个模块里面定义的变量与函数，默认为私有的，对其他模块不可见。一个模块要想对外暴露其内部的私有变量与函数，只能通过 module.exports 实现。

<!-- index.wxml -->
<view>{{price}}</view>
<!-- <view>¥36.80元</view> 目标 -->
<!-- <view>{{price.toFixed(2)}}</view> 不支持 -->

<wxs src="../../wxs/price.wxs" module="priceModule"></wxs>
<view>{{priceModule.priceFn(price)}}</view>
// index.js
Page({
  data: {
    price: 36.8,
  },
});
// priceFn.wxs
function priceFn(num) {
  return "¥" + num.toFixed(2) + "元";
  // return `¥${num.toFixed(2)}元`; // 不支持
}

module.exports = {
  priceFn: priceFn,
  // priceFn, // 不支持
};

十一. 数据缓存

11.1 wx.setStorageSync(string key, any data)

将数据存储在本地缓存中指定的 key 中，会覆盖掉原来该 key 对应的内容，是wx.setStorage的同步版本。

除非用户主动删除或因存储空间原因被系统清理，否则数据都一直可用。

单个 key 允许存储的最大数据长度为 1MB，所有数据存储上限为 10MB。

参数说明：

  • string key：本地缓存中指定的 key
  • any data：需要存储的内容。只支持原生类型、Date、及能够通过JSON.stringify序列化的对象
wx.setStorageSync('key1', 'hello')
wx.setStorageSync('key2', {num:1,str:'hello',boolean:true})

除非用户主动删除或因存储空间原因被系统清理，否则数据都一直可用。

单个 key 允许存储的最大数据长度为 1MB，所有数据存储上限为 10MB。

十二.自定义组件

开发者可以将页面内的功能模块抽象成自定义组件，以便在不同的页面中重复使用；也可以将复杂的页面拆分成多个低耦合的模块，有助于代码维护。自定义组件在使用时与基础组件非常相似。

12.1 介绍

12.1.1 创建自定义组件

类似于页面，一个自定义组件由 json wxml wxss js 4个文件组成。要编写一个自定义组件，首先需要在 json 文件中进行自定义组件声明（将 component 字段设为 true 可将这一组文件设为自定义组件）：

// components/child/child.json
{
  "component": true
}

同时，还要在 wxml 文件中编写组件模板，在 wxss 文件中加入组件样式，它们的写法与页面的写法类似。具体细节和注意事项参见 组件模板和样式 。

在自定义组件的 js 文件中，需要使用 Component() 来注册组件，并提供组件的属性定义、内部数据和自定义方法。

组件的属性值和内部数据将被用于组件 wxml 的渲染，其中，属性值是可由组件外部传入的。更多细节参见 Component构造器 。

// components/child/child.js
Component({
  /**
   * 组件的属性列表
   */
  properties: {
    otherText: {
      type: String,
      value: "goodbye",
    },
    otherText: String, // 简化的定义方式
  },

  /**
   * 组件的初始数据
   */
  data: {
    ownText: "hello",
  },
});

12.1.2 使用自定义组件

使用已注册的自定义组件前，首先要在页面的 json 文件中进行引用声明。此时需要提供每个自定义组件的标签名和对应的自定义组件文件路径：

pages/index/index.json文件：

{
  "usingComponents": {
    "child": "../../components/child/child"
  }
}

这样，在页面的 wxml 中就可以像使用基础组件一样使用自定义组件。节点名即自定义组件的标签名，节点属性即传递给组件的属性值。

<!-- pages/index/index.wxml -->
<view>这里是调用自定义组件的页面</view>
<child other-text="再见"></child>

12.2 组件模板

组件模板的写法与页面模板相同。组件模板与组件数据结合后生成的节点树，将被插入到组件的引用位置上。

在组件模板中可以提供一个 <slot> 节点，用于承载组件引用时提供的子节点。

12.2.1 模板数据绑定

与普通的 WXML 模板类似，可以使用数据绑定，这样就可以向子组件的属性传递动态数据。

12.2.2 组件 wxml 的 slot

在组件的 wxml中可以包含 slot 节点，用于承载组件使用者提供的 wxml 结构。

默认情况下，一个组件的 wxml 中只能有一个 slot 。

注意：微信小程序的slot不支持默认值！

<!-- components/child/child.wxml -->
<view>这里是组件的内部节点</view>
<view>{{ownText}}</view>
<view>{{otherText}}</view>
<slot></slot>
<!-- pages/index/index.wxml -->
<view>这里是调用自定义组件的页面</view>
<child other-text="再见">
  <view>这是通过slot添加的文本</view>
</child>

需要使用多 slot 时，必须在组件 js 中声明启用，在options字段中，将multipleSlots设置为true(注意这里不是在json文件中配置)：

Component({
  options: {
    multipleSlots: true // 在组件定义时的选项中启用多 slot 支持
  },
  properties: { /* ... */ },
  data: { /* ... */ },
  methods: { /* ... */ }
})

此时，可以在这个组件的 wxml 中使用多个 slot ，以不同的 name 来区分。

<!-- components/child/child.wxml -->
<view>这里是组件的内部节点</view>
<view>{{ownText}}</view>
<slot name="before"></slot>
<view>{{otherText}}</view>
<slot name="after"></slot>
// components/child/child.js
Component({
  options: {
    multipleSlots: true,
  },
  properties: {
    otherText: {
      type: String,
      value: "goodbye",
    },
  },
  data: {
    ownText: "hello",
  },
});

<!-- pages/index/index.wxml -->
<view>这里是调用自定义组件的页面</view>
<child other-text="再见">
  <view slot="before">先插入</view>
  <view slot="after">后插入</view>
</child>

12.2.3 组件样式

组件对应的 wxss 文件的样式，默认只对组件 wxml内的节点生效。编写组件样式时，需要注意：组件页面不能使用 id 选择器！

12.3 组件间通信与事件

组件间的基本通信方式有以下两种：

  • WXML 数据绑定：用于父组件向子组件的指定属性设置数据。
  • 事件：用于子组件向父组件传递数据，可以传递任意数据。

12.3.1 触发事件

自定义组件触发事件时，需要使用 triggerEvent 方法，指定事件名、detail对象和事件选项：

<!-- components/child/child.wxml -->
<button bindtap="handleTap">点击这个按钮将触发“myEvent”事件</button>
// components/child/child.js
Component({
  // 注意：自定义组件中的方法需要写在methods中，而page页面的js的方法则不需要
  methods: {
    handleTap() {
      // this.triggerEvent("myEvent", 'hello');
      // this.triggerEvent("myEvent", [1,2,3]);
      this.triggerEvent("myEvent", { a: 1, b: 2 }); // Vue的写法是this.$emit("myEvent", { a: 1, b: 2 })
    },
  },
});

12.3.2 监听事件

自定义组件可以触发任意的事件，引用组件的页面可以监听这些事件。监听自定义组件事件的方法与监听基础组件事件的方法完全一致：

<!-- pages/index/index.wxml -->
<!-- Vue的写法是@myEvent="handleMyEvent" -->
<!-- 这里也可以写成bindmyEvent="handleMyEvent" -->
<child bind:myEvent="handleMyEvent"></child>
// pages/index/index.js
Page({
  handleMyEvent(e) {
    console.log(e);
    const { a, b } = e.detail;
    console.log(a, b);
  },
});

自定义组件仍然可以拥有id属性和data-xx属性，属性集合仍然在事件对象的currentTarget和target属性集合里：

<!--pages/index/index.wxml-->
<child id="1" data-id="1" bind:myEvent="handleMyEvent"></child>

也可以直接在自定义组件标签上绑定微信小程序原生事件，属性集合仍然在事件对象的currentTarget和target属性集合里，但是此时detail事件对象里没有子组件传递给父组件的参数：

<!--pages/index/index.wxml-->
<child id="1" data-id="1" bind:tap="onTap"></child>

与Vue的子组件触发事件做对比：

子组件触发事件：

// App.vue
<template>
  <div>
    <child @myEvent="handleClick"></child>
  </div>
</template>

<script>
 import Child from './Comp.vue'
	export default{
    components:{
      Child
    },
    methods:{
      handleClick(a,b,c){
        console.log(a,b,c)
      }
    }
  }
</script>
// Comp.vue
<template>
  <button @click="handleClick">
    child
  </button>
</template>

<script>
	export default{
    methods:{
      handleClick(){
        this.$emit('myEvent',1,2,3)
      }
    }
  }
</script>

12.3.3 组件事件阻止冒泡

<!--components/child.wxml-->
<view catchtap="handleTap">
  <view>child</view>
</view>
// components/child.js
Component({
  methods: {
    handleTap() {
      this.triggerEvent("myEvent", "child");
    },
  },
});
<!--index.wxml-->
<view bindtap="handleParent">
  <child bind:myEvent="handleChild"></child>
</view>
// index.js

Page({
  handleChild() {
    console.log("child");
  },
  handleParent() {
    console.log("parent");
  },
});

12.4 组件生命周期

组件的生命周期，指的是组件自身的一些函数，这些函数在特殊的时间点或遇到一些特殊的框架事件时被自动触发。

12.4.1 定义组件生命周期

生命周期方法可以直接定义在 Component 构造器的第一级参数中。自小程序基础库版本 2.2.3 起，组件的的生命周期也可以在 lifetimes 字段内进行声明（这是推荐的方式，其优先级最高）。

Component({
  lifetimes: {
    created() {
      // 组件实例刚刚被创建好时执行
    },
    attached() {
      // 在组件实例进入页面节点树时执行
    },
    detached() {
      // 在组件实例被从页面节点树移除时执行
    },
  },
  // 以下是旧式的定义方式，可以保持对 <2.2.3 版本基础库的兼容
  created() {
      // 组件实例刚刚被创建好时执行
  },
  attached() {
    // 在组件实例进入页面节点树时执行
  },
  detached() {
    // 在组件实例被从页面节点树移除时执行
  },
  // ...
})

组件实例刚刚被创建好时， created 生命周期被触发。此时可以访问this.data ，但是此时还不能调用 setData 。

在组件完全初始化完毕、进入页面节点树后， attached 生命周期被触发，此时可以调用 setData 。 

// components/child/child.js
Component({
  lifetimes: {
    created() {
      // 组件实例刚刚被创建好时执行
      console.log("created");
      console.log(this.data);
      
      // 无效
      this.setData({
        ownText: "hi",
      });
    },
    attached() {
      // 在组件实例进入页面节点树时执行
      console.log("attached");
      
      // 有效
      this.setData({
        ownText: "hi",
      });
    },
    detached() {
      // 在组件实例被从页面节点树移除时执行
      console.log("detached");
    },
  }
});

在组件离开页面节点树后， detached 生命周期被触发：

<!-- pages/index/index.wxml -->
<child wx:if="{{showChild}}"></child>
<button bindtap="toggleChild">控制child显示隐藏</button>
// pages/index/index.js
Page({
  data: {
    showChild: true,
  },
  toggleChild() {
    this.setData({
      showChild: !this.data.showChild,
    });
  },
});

12.4.2 学过的生命周期

到这里基本我们学完了重要的生命周期，总结一下：

全局生命周期：

// app.js
App({
  onLaunch() {
    // 监听小程序初始化，全局只触发一次
    console.log("onLaunch");
  },
  onShow() {
    // 监听小程序启动或切前台
    console.log("onShow");
  },
  onHide() {
    // 监听小程序切后台 
    console.log("onHide");
  },
});

页面生命周期：

// page.js
Page({
  // 监听页面加载
  onLoad(options) {
    console.log("onLoad");
  },
  // 监听页面初次渲染完成
  onReady() {
    console.log("onReady");

    console.log("onHide");

  // 页面卸载时触发。如wx.redirectTo、wx.navigateBack、wx.reLaunch到其他页面时。
  onUnload() {
    console.log("onUnload");
  },
});
